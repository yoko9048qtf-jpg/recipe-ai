# database.md

## 結論: このプロジェクトは永続データベースを一切使用していない

コードベース全体（`server/`, `client/`, `api/`）を検索した結果、SQL/NoSQL クライアント、ORM
（Prisma, Sequelize, Mongoose等）、マイグレーションファイル、スキーマ定義ファイルは存在しない。
`package.json`（ルート/client/server いずれも）にもDB関連の依存パッケージは含まれていない。

## 代わりに使われている状態保持の仕組み

| 種類 | 保存場所 | 内容 | 永続性 |
|---|---|---|---|
| カテゴリ一覧キャッシュ | `server/rakuten.js` の `_catCache`（モジュールスコープ変数） | 楽天レシピの大カテゴリ一覧 | プロセスが生きている間のみ。再起動で消える |
| ランキングキャッシュ | `server/rakuten.js` の `_rankCache`（`Map<categoryId, {at, recipes}>`） | カテゴリ別ランキング、TTL 1時間 | 同上（サーバーレス環境では実行毎にリセットされる可能性が高いとREADMEに明記） |
| 直近表示レシピ履歴 | ブラウザの `localStorage`（キー: `recipeHistory`） | 表示済みレシピIDの配列（最大10件） | ブラウザ・端末単位で永続。サーバー側には一切保存されない |
| 画面状態・入力内容 | React state（`App.tsx` ほか） | 選択食材・ジャンル・人数・現在の画面等 | ページリロードで消える（意図的な設計） |

## なぜDBが無いのか（推測）

- レシピデータそのものは楽天レシピAPIが情報源であり、自前で複製・永続化する必要がない
- ユーザーアカウント・お気に入り保存・レビュー投稿など「DBが要る機能」がそもそも実装されていない
  （[requirements.md](./requirements.md) 参照）
- 個人〜家庭利用規模を想定しており、サーバーレス環境（Vercel）へのデプロイのしやすさを優先し、
  ステートレスな構成にしていると考えられる

## 今後DBが必要になりうるケース（改善提案・推測）

- お気に入りレシピの保存、複数端末間での履歴同期を実装する場合
- 楽天APIのレート制限を回避するための永続キャッシュ（README内で「Vercel KV等の外部キャッシュ導入を検討」と
  既に言及あり）
- ユーザーごとの食材在庫管理（在庫を都度チェックし直す現行UXから、保存された在庫を編集するUXへ変える場合）

これらは現時点では実装されておらず、[todo.md](./todo.md) の「未実装」に位置づける。
